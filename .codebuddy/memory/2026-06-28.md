# 2026-06-28

## 标签卡片布局修复

修复了持续许久的标签卡片第二行拉伸问题（第二行的卡片变成长方形占满整行）。

### 根因分析
两个问题叠加：
1. **CSS grid 列宽用了 `1fr`** — `grid-template-columns: repeat(auto-fill, minmax(70px, 1fr))`。当第二行只有 1-2 个标签时，`1fr` 会让它们拉伸占满整行宽度。
2. **JS 中每张卡片多输出了一个 `</div>`** — `renderGrid()` 函数里第 96 行有一个孤立的 `html+='</div>';`，虽然浏览器会忽略多余闭合标签，但属于 HTML 结构错误。

### 修改内容
- `static/style.css`: `.tag-grid` 列宽从 `repeat(auto-fill,minmax(70px,1fr))` 改为 `repeat(auto-fill,85px)`，固定列宽，避免拉伸。后来调到 `100px` 以增大卡片。

## 标签卡片大小/图片存储/浏览全部弹窗

### 修改清单
1. **卡片大小** — grid列宽从 `85px` 改为 `100px`（`style.css`）
2. **标签图片独立存储** — 从 `init()` 中移除 `_rebuildTagImgIndex()` 调用，`_tagImgIndex` 不再依赖返图区 `_galleryImages`。标签图片索引纯 localStorage 存储，清空返图区不受影响。
3. **删除标签时也删除关联图片** — `delTagClick()` 中添加 `delete _tagImgIndex[en]` 并保存到 localStorage。
4. **浏览全部按钮** — `renderGrid()` 中将 "🗑 批量删除" 按钮替换为 "📋 浏览全部" 按钮，打开新弹窗展示当前子类全部标签的纯文字列表（含勾选框）。
### 删除标签时智能处理关联图片

**逻辑**：删除标签时先遍历该标签关联的所有图片，对每张 `localUrl` 图片：
1. 检查该图片 URL 是否被其他标签引用 → 如果是，跳过文件删除
2. 检查图片是否还在 `_galleryImages`（返图区）中 → 如果是，跳过文件删除
3. 检查图片是否在任意相册中 → 如果是，跳过文件删除
4. 只有 **无其他标签引用 + 不在返图区 + 不在相册** 的图片，才调用 `POST /api/tag-img/delete` 删除 `tag_assets` 中的文件
5. 最后从 `_tagImgIndex` 中移除整个标签条目

**涉及文件**：
- `server.py`: 新增 `POST /api/tag-img/delete` 端点
- `static/app.js`: `delTagClick()` + 批量删除均使用上述智能逻辑

### Bug 修复：tag-img/save 死锁导致图片保存失败

**问题**：`/api/tag-img/save` 端点原来通过 `http://127.0.0.1:5802/api/comfyui/proxy-image?...` 自调用 Flask 来下载图片。Flask 默认单线程，save 请求阻塞等待 proxy-image 响应，但服务器正忙于处理 save 请求无法处理 proxy-image 请求 → **死锁** → 30秒超时后静默失败 → `localUrl` 永远没设上 → 删除返图后标签卡片无图片。

**修复**：
1. `server.py`: `/api/tag-img/save` 改为**直接请求 ComfyUI**（`{COMFYUI_URL}/view?filename=...`），不走 Flask 代理。
2. `server.py`: `app.run(...)` 添加 `threaded=True`，防止其他自调用场景死锁。
3. `static/app.js`: `_updateTagImgIndex` 中新增图片时立即调用 `_saveTagImageForTagIndex` 保存，不等删除时才保存。添加 `_savingTagUrls` 去重。
4. `static/app.js`: 新增 `_tagCardImgError()` 函数。标签卡片图片加载失败时自动触发 `_saveTagImageForTagIndex`，保存成功后重新渲染。兼容修复前创建的旧标签关联（没有 localUrl 的旧数据也能自动补上）。

### Bug 修复：批量删除导致 JSON 文件损坏
批量删除原来用 `forEach` 并发发送所有删除请求，多个并发写入操作同一 JSON 文件产生 race condition。
- `data/tags.json` 被写损坏（末尾多出额外 JSON 片段）
- `data/custom_tags.json` 被清空

修复方案：
1. **`static/app.js`**: `_bindBrowseModal()` 中批量删除改为 **串行递归**（`deleteNext()`），逐个发送请求，完成后才发下一个。
2. **`server.py`**: `write_json()` 改为先写入 `*.tmp` 临时文件，再用 `os.replace()` 原子替换，防止写入中断导致文件损坏。
3. **`data/tags.json`**: 从 git 恢复（未被损坏的版本）。
4. **`data/custom_tags.json`**: 本来就是空的 `{"categories":[]}`，无数据丢失。
- `static/app.js`: `renderGrid()` 函数中删除了多余的 `html+='</div>';`（原第 96 行），确保每张卡片的 HTML 结构正确闭合。

### 标签图片顺序与淘汰清理

**发现的问题**：
1. `_pollComfyUI` 用 `for(var i=r.images.length-1;i>=0;i--)` 反向遍历 + `unshift`，导致最早生成的图在数组第 0 位 → 封面不是最新图。改为正向遍历。
2. `_openTagGallery` 写死了 `img.url`，未用 `img.localUrl || img.url` → 返图区删除后图片 404。已修复。
3. `_updateTagImgIndex` 超出 3 张时直接 `arr.length=3` 静默丢弃，没清理 tag_assets 文件。

**新增的淘汰清理逻辑**（在 `_updateTagImgIndex` 中）：
- 第 4 张旧图被淘汰时，检查其 `localUrl`：
  - 若有其他标签引用同一 URL → 不删文件
  - 若仍在返图区 `_galleryImages` → 不删文件
  - 若仍在任意相册 → 不删文件
  - 都不满足 → 调用 `POST /api-tag-img/delete` 删除 `tag_assets/` 中的文件
- 然后 `arr.length=3` 截断
- 新图片 `unshift` 到数组头部 → 最新图片始终在封面
